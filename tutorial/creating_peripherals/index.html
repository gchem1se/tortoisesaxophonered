    Adding a Custom Peripheral to a SoC in QEMU: a practical example

    The following steps will guide you through configuring and implementing the peripheral within QEMU by using a practical example. In our project, we implemented a custom emulation of the LPUART peripheral model of the NXP S32K3X8_EVB board.

    One of the most important challenges in designing this peripheral is deciding how deep the emulation should go. Should it only provide basic register access and pass-through functionality, or should it fully replicate the real hardware’s internal logic, including timeouts, interrupts, and FIFO behavior? The right level of emulation depth depends on the intended use case - whether for software development, debugging, or full-system validation.

    In our example, our scope has been to grant these basic functionalities of the peripheral:

    • register handling and reset
    • Interrupt management
    • character device interaction (transmission and reception of bytes)

    Implementing the Peripheral (s32k3x8_usart.h)

    In our implementation, we kept the same approach we exploited when adding support for the machine itself. In particular, we used a header file (qemu/include/hw/char/s32k3x8_lpuart.h) to store constants, such as base addresses and reset values we found in the reference manual on the vendor's site.

    // [...]
#define STAT 0x14    /**< Status */
#define CTRL 0x18    /**< Control */
#define DATA 0x1C    /**< Data */
// [...]

// [...]
#define LPUART_STAT_RESET 0x00C00000
#define LPUART_CTRL_RESET 0x00000000
#define LPUART_DATA_RESET 0x00001000
// [...]

// [...]
#define CTRL_TE (1 << 19)    // You can enable the transmitter by writing 1 here
#define CTRL_RE (1 << 18)    // You can enable the receiver by writing 1 here
#define STAT_TDRE (1 << 23)  // if 1, another character may be written to the transmit FIFO
#define STAT_TC (1 << 22)    // if 0, the transmitter is active; if 1, the transmitter is idle
#define STAT_RDRF (1 << 21)
// [...]

    Other than these declaration, as we already saw in the creation of the board, the header should declare the State structure of the UART peripheral, so it needs to store all the registers and IRQ lines if present, and a MMIO region to be mapped in the system memory of the SoC later on.

    Given that we're operating on a character-based peripheral, and QEMU already offers some primitives to handle that, we have to include in the state a CharBackend object. 

    struct S32K3X8LpuartState
{
    /* <private> */
    SysBusDevice parent_obj;

    /* <public> */
    MemoryRegion mmio;

    // [...]
    uint32_t lpuart_stat;
    uint32_t lpuart_ctrl;
    uint32_t lpuart_data;
    // [...]

    CharBackend chr;
    qemu_irq irq;
};

    Implementing the Peripheral (s32k3x8_usart.c)

    The new UART peripheral should be implemented as a C source file inside qemu/hw/char/. The file should include:

    • Initialization functions: Define how the peripheral is instantiated and registered in QEMU.
    • Read/Write functions: Handle access to memory-mapped registers.
    • Interrupt handling: Manage IRQs triggered by the peripheral.
    • QEMU integration: Use QEMU’s internal macros and functions for device emulation.

    The following header files should be included:

    #include "qemu/osdep.h"
#include "hw/char/s32k3x8_lpuart.h"
#include "hw/irq.h"
#include "hw/qdev-properties.h"
#include "hw/qdev-properties-system.h"
#include "qemu/log.h"
#include "qemu/module.h"

    A sample implementation should include functions for:

    • Device initialization and realization within QEMU (s32k3x8_usart_init, s32k3x8_usart_realize, s32k3x8_usart_class_init)
      • s32k3x8_usart_init also registers memory-mapped I/O (MMIO) and the IRQ line.
    • Checking if the peripheral can receive data (s32k3x8_usart_can_receive)
    • Receiving data (s32k3x8_usart_receive)
    • Resetting the peripheral (s32k3x8_usart_reset)
    • Handling register reads (s32k3x8_usart_read)
    • Handling register writes (s32k3x8_usart_write)

    These functions have to be implemented in according with how the datasheet / reference manual states must adhere strictly to the behavior and specifications outlined in the official S32K3X8 LPUART datasheet.

    Defining Memory-Mapped Registers

    After registering and initializing the MMIO memory region of the peripheral in the init function, the peripheral should also expose registers through a memory-mapped interface. This is done using MemoryRegionOps in QEMU:

    static const MemoryRegionOps s32k3x8_usart_ops = {
    .read = s32k3x8_usart_read,
    .write = s32k3x8_usart_write,
    .endianness = DEVICE_NATIVE_ENDIAN,
};

    read and write functions

    The read and write functions handle access to registers based on the given offset (addr). These functions update internal state variables (in our case, the values of the three most important registers: STAT, DATA and CTRL), and trigger appropriate actions, such as setting flags or sending interrupts.

    Registering the peripheral Type in QEMU

    Finally, the peripheral must be registered as a QEMU type so it can be instantiated in a SoC.

    static const TypeInfo s32k3x8_usart_info = {
    .name          = TYPE_S32K3X8_LPUART,
    .parent        = TYPE_SYS_BUS_DEVICE,
    .instance_size = sizeof(S32K3X8LpuartState),
    .instance_init = s32k3x8_usart_init,
    .class_init    = s32k3x8_usart_class_init,
};

static void s32k3x8_usart_register_types(void) {
    type_register_static(&s32k3x8_usart_info);
}

 type_init(s32k3x8_usart_register_types)

    This process ensures that the peripheral is recognized as a valid QEMU device and can be instantiated in an emulated SoC.
